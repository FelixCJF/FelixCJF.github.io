设计模式之观察者模式

Java设计模式
  • 观察者模式

定义：一种对象间一对多的依赖关系，当一个对象状态发送改变时，依赖它的对象都将会受到通知，并发生改变。行为类模式

类图

在软件需求分析的时候经常会出现这样一种依赖的关系：当一个对象作出了改变的时候，其他依赖这个对象的对象也不得不作出相应的改变。比如生活中，我们订阅的天气预报短信，发给不同的人，不同的人便会对此作出不同的改变。不同的人观察天气情况，所以不同的人便是观察者的角色。

观察者模式结构

观察者： 一般是一个接口，只有一个update方法，当被观察者状态发生改变的时候，更新方法就会被触发调用。
被观察者： 被观察者有一个容器用来放多个观察者对象，所以是多个观察者依赖一个被观察者的关系。被观察者的attach方法作用是忘容器中添加观察者对象。detach是删除观察者对象。notify方法是当被观察者状态发生改变时，调用观察者对象的对应方法。
具体的观察者：为了便于扩展，在观察者角色中定义具体的业务逻辑。
具体的被观察者：存储目标的具体状态，定义状态发生改变时要处理的业务逻辑。

观察者模式实现模型

拉模型

被观察者不知道观察者所需要的具体消息，不管观察者是否需要，把所有相关的信息或数据全部推送给观察者

推模型

被观察者知道观察者所需要的具体信息，只会把少量有用的信息给观察者

观察者模式通用代码实现

  1. 观察目标对象的定义 

    public class Subject {
//观察者容器
private List<Observer> observers = new ArrayList<Observer>();

public void attach(Observer o){
    observers.add(o);
}
public void detach(Observer o) {
    observers.remove(o);
}
protected void notifyObserver(){
    for (Observer observer : observers) {
        observer.update(this);
    }
}
}

  1. 具体目标对象定义

    public class ConcreteSubject extends Subject {

//目标对象的状态
private String state;

public String getState() {
    return state;
}

public void setState(String state) {
    this.state = state;
    this.notifyObserver();
}    
}

  2. 观察者接口定义

    public interface Observer {
/**
 * 更新接口
 * @param subject 传入的具体目标类
 */
public void update(Subject subject);
}

  3. 观察者具体实现

`

public class ConcreteObserver implements Observer {
//观察者状态
private String observerState;

/**
 * 获取目标类的状态，同步到观察者中
 */
public void update(Subject subject) {
    observerState = ((ConcreteSubject)subject).getState();
}
}`

用Java提供对观察者模式的支持

java提供观察者模式支持的优点

  • 不需要自己定义目标类和观察者接口了
  • 不需要维护注册信息了
  • 状态改变触发需要调用setchange()方法
  • 可以同时实现推模型和拉模型

    观察者模式实现天气查询

  1. 天气目标类 
public class ConcreteWeatherSubject extends Observable {
//天气情况
private String content;
public String getContent() {
    return content;
}

public void setContent(String content) {
    this.content = content;
    //通知所有观察者天气情况，通知之前必须先调用setchanged（）
    this.setChanged();
    //推模式
    this.notifyObservers(content);
    //拉模式
    //this.notifyObservers();
}    
}

  1. 观察者类

    public class ConcreteWeatherObserver implements Observer {

//观察者名称
private String ObserverName;

public void update(Observable o, Object arg) {
    //推方式
    System.out.println(ObserverName + "收到了消息，目标推送了内容：" + arg);
    //拉模式
    System.out.println(ObserverName + "收到了消息，目标拉的内容是：" + ((ConcreteWeatherSubject)o).getContent());;
}

public String getObserverName() {
    return ObserverName;
}

public void setObserverName(String observerName) {
    ObserverName = observerName;
}    
}

  2. 测试类 

    @Test
public void WeatherTest(){
//创建一个天气类作为被观察者
ConcreteWeatherSubject subject = new ConcreteWeatherSubject();

//创建观察者小明
ConcreteWeatherObserver boy = new ConcreteWeatherObserver();
boy.setObserverName("小明");

//创建观察者小红
ConcreteWeatherObserver girl = new ConcreteWeatherObserver();
girl.setObserverName("小红");

//注册观察者
subject.addObserver(boy);
subject.addObserver(girl);

//天气更新
subject.setContent("天气晴");
}`

    观察者模式的优缺点

    优点

  • 实现了观察者和目标类之间的抽象耦合
  • 实现了动态联动
  • 可以实现广播通信

    缺点

  • 链式触发，性能堪忧
  • 容易做无谓的操作

总结

观察者模式衍生出一些问题，当观察者不需要被观察者发生任何改变都发送通知，只需要对自己有影响的改变时才发出通知然后更新。那么就需要在发送通知前进行判断。
